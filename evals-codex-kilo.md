1. HIGH-LEVEL ARCHITECTURE (Grounded only)
- FastAPI service bootstraps the API app, CORS, and rate limiting, and initializes DB tables on startup (`airopa_automation/api/main.py:app`, `airopa_automation/api/main.py:startup_event`, `airopa_automation/api/rate_limit.py:limiter`, `airopa_automation/api/models/database.py:init_db`).
- Health endpoint checks DB connectivity and reports status (`airopa_automation/api/routes/health.py:health_check`, `airopa_automation/api/models/database.py:get_db`).
- Articles read API lists and fetches article records from the SQLAlchemy `articles` table with filtering and pagination (`airopa_automation/api/routes/articles.py:list_articles`, `airopa_automation/api/routes/articles.py:get_article`, `airopa_automation/api/models/database.py:Article`).
- Jobs/scrape API creates scrape jobs, runs the pipeline asynchronously or synchronously, and returns job status; endpoints require API key auth (`airopa_automation/api/routes/jobs.py:trigger_scrape`, `airopa_automation/api/routes/jobs.py:trigger_scrape_sync`, `airopa_automation/api/routes/jobs.py:get_job_status`, `airopa_automation/api/auth.py:verify_api_key`, `airopa_automation/api/models/database.py:Job`).
- Pipeline service runs the scrape -> classify -> summarize -> quality-score -> store flow and persists telemetry/metrics (`airopa_automation/api/services/pipeline.py:PipelineService.run_scrape_job`, `airopa_automation/api/services/pipeline.py:_store_article`, `airopa_automation/api/services/pipeline.py:_record_telemetry`, `airopa_automation/api/services/pipeline.py:_record_source_metrics`).
- Core agents implement scraping, classification, summarization, quality scoring, markdown generation, and git commits (`airopa_automation/agents.py:ScraperAgent`, `airopa_automation/agents.py:CategoryClassifierAgent`, `airopa_automation/agents.py:SummarizerAgent`, `airopa_automation/agents.py:QualityScoreAgent`, `airopa_automation/agents.py:ContentGeneratorAgent`, `airopa_automation/agents.py:GitCommitAgent`).
- LLM access is abstracted and validated by wrappers and schema parsers (`airopa_automation/llm.py:llm_complete`, `airopa_automation/llm_schemas.py:parse_classification`, `airopa_automation/llm_schemas.py:parse_summary`).
- Persistence layer uses SQLAlchemy models for `articles`, `jobs`, `source_metrics`, and `llm_telemetry`, and builds the engine from `DATABASE_URL` (`airopa_automation/api/models/database.py:Article`, `airopa_automation/api/models/database.py:Job`, `airopa_automation/api/models/database.py:SourceMetric`, `airopa_automation/api/models/database.py:LLMTelemetry`, `airopa_automation/api/models/database.py:engine`).
- Scheduled execution exists via GitHub Actions hitting the synchronous scrape endpoint and a direct cron script that runs the pipeline service (`.github/workflows/scheduled-scrape.yml`, `airopa_automation/api/routes/jobs.py:trigger_scrape_sync`, `scripts/cron_scrape.py:main`).
- A standalone CLI pipeline can scrape, classify, generate markdown, and commit content to git (`main.py:AutomationPipeline`, `main.py:main`, `airopa_automation/agents.py:ContentGeneratorAgent`, `airopa_automation/agents.py:GitCommitAgent`).

2. DATAFLOW WALKTHROUGH
1) A client triggers a scrape via `POST /api/scrape/sync`, which is rate-limited and API-key protected (`airopa_automation/api/routes/jobs.py:trigger_scrape_sync`, `airopa_automation/api/auth.py:verify_api_key`, `airopa_automation/api/rate_limit.py:SCRAPE_RATE_LIMIT`).
2) The API creates a `jobs` record with status `queued` and persists it (`airopa_automation/api/routes/jobs.py:trigger_scrape_sync`, `airopa_automation/api/models/database.py:Job`).
3) The request runs the pipeline service to process the job (`airopa_automation/api/routes/jobs.py:trigger_scrape_sync`, `airopa_automation/api/services/pipeline.py:PipelineService.run_scrape_job`).
4) The pipeline scrapes RSS and web sources into `Article` objects, then deduplicates them (`airopa_automation/api/services/pipeline.py:PipelineService.run_scrape_job`, `airopa_automation/agents.py:ScraperAgent.scrape_rss_feeds`, `airopa_automation/agents.py:ScraperAgent.scrape_web_sources`, `airopa_automation/api/services/pipeline.py:PipelineService._remove_duplicates`).
5) Articles are classified, optionally using LLM calls and telemetry capture, with keyword fallback (`airopa_automation/api/services/pipeline.py:PipelineService.run_scrape_job`, `airopa_automation/agents.py:CategoryClassifierAgent.classify`, `airopa_automation/llm.py:llm_complete`, `airopa_automation/llm_schemas.py:parse_classification`).
6) Articles are summarized with optional LLM calls and telemetry capture (`airopa_automation/api/services/pipeline.py:PipelineService.run_scrape_job`, `airopa_automation/agents.py:SummarizerAgent.summarize`, `airopa_automation/llm.py:llm_complete`, `airopa_automation/llm_schemas.py:parse_summary`).
7) Articles are quality-scored and filtered to a high-quality subset (`airopa_automation/api/services/pipeline.py:PipelineService.run_scrape_job`, `airopa_automation/agents.py:QualityScoreAgent.assess_quality`).
8) High-quality articles are stored in the `articles` table and per-run telemetry/metrics are written (`airopa_automation/api/services/pipeline.py:PipelineService._store_article`, `airopa_automation/api/services/pipeline.py:_record_telemetry`, `airopa_automation/api/services/pipeline.py:_record_source_metrics`, `airopa_automation/api/models/database.py:Article`, `airopa_automation/api/models/database.py:LLMTelemetry`, `airopa_automation/api/models/database.py:SourceMetric`).
9) The job status is updated to `completed` with result counts, or `failed` on error (`airopa_automation/api/services/pipeline.py:PipelineService.run_scrape_job`, `airopa_automation/api/models/database.py:Job`).
10) Clients retrieve stored items via `GET /api/articles` and `GET /api/articles/{id}` (`airopa_automation/api/routes/articles.py:list_articles`, `airopa_automation/api/routes/articles.py:get_article`).

3. DECISION RECORD (Why)
- What: Use FastAPI for the API layer; Why: UNKNOWN (no explicit rationale found); Evidence: `airopa_automation/api/main.py:app`; Alternatives: INFERRED - Flask, Django, Starlette; Trade-offs: INFERRED - FastAPI adds async tooling and type validation overhead but improves docs and speed.
- What: Enforce IP-based rate limiting via slowapi; Why: "limit request rates and prevent abuse" is stated; Evidence: `airopa_automation/api/rate_limit.py` module docstring and `limiter`; Alternatives: INFERRED - API gateway throttling, NGINX rate limiting; Trade-offs: INFERRED - per-instance state can be inconsistent without shared storage.
- What: Protect scrape endpoints with X-API-Key; Why: "Provides API key authentication for protected endpoints" is stated; Evidence: `airopa_automation/api/auth.py:verify_api_key`, `airopa_automation/api/routes/jobs.py:trigger_scrape`; Alternatives: INFERRED - OAuth, JWT; Trade-offs: INFERRED - API keys are simpler but coarse-grained and harder to rotate.
- What: Use SQLAlchemy models with DATABASE_URL and postgres URL normalization; Why: comment notes Railway’s `postgres://` requires normalization; Evidence: `airopa_automation/api/models/database.py:DATABASE_URL` and replacement logic; Alternatives: INFERRED - raw SQL, other ORMs; Trade-offs: INFERRED - ORM abstraction vs. direct SQL control.
- What: Gate LLM features via feature flags and shadow mode; Why: "Feature flags - control LLM rollout" is stated; Evidence: `airopa_automation/config.py:AIConfig` comment, `airopa_automation/agents.py:CategoryClassifierAgent.classify` docstring; Alternatives: INFERRED - always-on LLM or offline batch processing; Trade-offs: INFERRED - more complexity but safer rollout.
- What: Apply a per-run token budget to LLM usage; Why: "Prevents runaway costs" is stated; Evidence: `airopa_automation/budget.py` module docstring, `TokenBudget`; Alternatives: INFERRED - no budget, per-request caps only; Trade-offs: INFERRED - may skip LLM quality when budget is exhausted.

- What: Schedule scrapes every 6 hours via GitHub Actions calling `/api/scrape/sync`; Why: INFERRED - periodic refresh cadence; Evidence: `.github/workflows/scheduled-scrape.yml` (cron), `airopa_automation/api/routes/jobs.py:trigger_scrape_sync`; Alternatives: INFERRED - Railway cron, external scheduler, background worker; Trade-offs: INFERRED - relies on API availability and GitHub Actions uptime.

4. ISSUES AND RISKS
- Symptom: Protected endpoints become publicly accessible when `API_KEY` is unset; Location: `airopa_automation/api/auth.py:verify_api_key`; Impact: INFERRED - unauthorized callers can trigger `/api/scrape` and consume resources; Severity: S2; Fix sketch: require `API_KEY` at startup or return 503/401 when missing in non-dev environments.
- Symptom: `eu_relevance` and `confidence` are stored as `None` when their value is `0.0` due to truthiness checks; Location: `airopa_automation/api/services/pipeline.py:_store_article`; Impact: INFERRED - low-relevance items may bypass API filters that treat `NULL` as allowed (`airopa_automation/api/routes/articles.py:list_articles`); Severity: S2; Fix sketch: use `is not None` checks and store `0.0` explicitly.
- Symptom: Source metrics drop zero values from averages and counts due to truthiness checks; Location: `airopa_automation/api/services/pipeline.py:_record_source_metrics`; Impact: INFERRED - `avg_eu_relevance` and `avg_quality_score` can be biased upward, and relevance counts may be inaccurate; Severity: S3; Fix sketch: include zero values by checking `is not None` instead of truthiness.
- Symptom: RSS fetches do not set an explicit timeout; Location: `airopa_automation/agents.py:ScraperAgent.scrape_rss_feeds`; Impact: INFERRED - slow or hanging feeds can stall the pipeline; Severity: S2; Fix sketch: fetch feeds with `requests.get(..., timeout=...)` and pass content into `feedparser.parse`.
